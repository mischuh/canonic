title: "What CI checks"
description: "The two jobs that run on every push to main and every pull request."
---

canonic's CI is a single GitHub Actions workflow, `.github/workflows/ci.yml`, with two jobs. Both trigger on push to `main` and on every pull request.

<Note>
  This page documents exactly what's in `ci.yml` today. There is no separate release-automation pipeline (version bumping, changelog generation, package publishing) on `main` yet.
</Note>

## `check`

The standard lint/type/test gate:

```yaml
- uv sync --frozen --group dev
- uv run ruff check .           # lint
- uv run ruff format --check .  # format check
- uv run mypy canonic/          # type check
- uv run pytest                 # full test suite
```

This is the same sequence documented in the project's own contribution conventions; see [Running checks locally](/ci-cd/running-checks-locally) to run it yourself before pushing.

## `release-gate`

A second, separate job that runs the `release_gate`-marked test suite:

```yaml
- uv run pytest -m release_gate --strict-markers -rs --junitxml=release-gate.xml
```

Tests marked `release_gate` are conformance-gate tests: they must **pass**, not just run, before beta invites go out. The job adds an explicit check on top of pytest's own result:

```bash
if grep -q '<skipped' release-gate.xml; then
  echo "::error::Release-gate test(s) were skipped (e.g. Docker unavailable): gate cannot pass silently."
  exit 1
fi
```

This closes a gap plain pytest exit codes don't cover: a skipped test (for example because Docker wasn't available in the runner) normally doesn't fail the job. For this specific marker, that's treated as a hard failure; the gate is designed to never pass silently via skips. See [The release gate](/ci-cd/release-gate) for what these tests actually check.
